# Product Requirement Document: Project "Lumina"

**Version:** 1.0 (Draft)
**Date:** March 3, 2026
**Target Platform:** .NET 10 / C# 14
**Status:** In Design

## 1. Executive Summary

**Lumina** is a high-performance, append-only log storage and retrieval engine designed for observability and event streaming. It utilizes a tiered storage architecture: a crash-resilient, memory-mapped **Write-Ahead Log (WAL)** for hot ingestion, and column-oriented **Parquet** files for cold storage and analytics.

The system is designed to handle high-throughput ingestion from mixed sources (OTLP, JSON), survive catastrophic process failures without data loss, and provide near real-time search capabilities via an embedded OLAP engine (DuckDB).

## 2. Terminology

* **Stream:** A logical grouping of logs (e.g., `app-backend-prod`).
* **WAL (Hot Storage):** An append-only binary file optimized for write speed and durability.
* **Parquet (Cold Storage):** Compressed, columnar files optimized for read/scan speed.
* **Compactor:** A background process that transforms WAL entries into Parquet files.
* **Normalization:** The process of converting diverse input formats (JSON, OTel) into a unified internal binary format at the edge.
* **Sync Marker:** A unique byte sequence used to locate valid log boundaries after corruption.

## 3. Functional Requirements

### 3.1. Ingestion (The Head)

* **Protocols:**
  * **HTTP/3 (QUIC):** Standard JSON POST (Single entry & Batch).
  * **gRPC/Protobuf:** OpenTelemetry (OTLP) Logs Export Service.
* **Stream Isolation:** Clients must specify a `stream` identifier in the URL or header.
* **Normalization:** The system acts as a "funnel." Regardless of input format (JSON or Protobuf), data is immediately parsed, validated, and serialized into the internal **Unified Binary Format** before disk I/O.
* **Response:** Clients receive `200 OK` only after the data is successfully persisted to the WAL (fsync/WriteThrough).

### 3.2. Storage Engine (The Heart)

* **Immutability:** Once written to WAL, data is never modified.
* **Versioning:** The storage engine must support reading older WAL versions even after the binary format evolves.
* **Tiering:**
  * **L0:** Active WAL (RAM/Disk).
  * **L1:** Micro-batches (Parquet, 10-minute intervals).
  * **L2:** Consolidated (Parquet, Daily/Weekly with ZSTD compression).

### 3.3. Search & Retrieval (The Brain)

* **Query Interface:** REST API accepting SQL-like parameters or direct SQL.
* **Engine:** Embedded DuckDB .NET integration.
* **Scope:** Queries must be scoped to a specific `Stream` (and optional time range).
* **Schema on Read:** The system must handle "sparse" schemas where keys appear in some logs but not others.

## 4. Technical Architecture & Specifications

### 4.1. The Resilient WAL Format

To ensure data integrity and "intelligent re-sync" capabilities, the WAL will use a **Header-First** protection scheme with **Sync Markers**.

**File Extension:** `.wal`
**Byte Order:** Little-Endian

#### 4.1.1. File Header (Once per file)

| Offset | Size | Name | Description |
| :--- | :--- | :--- | :--- |
| 0 | 4B | Magic | `0x4C 0x55 0x4D 0x49` ("LUMI") |
| 4 | 1B | Version | Format Version (e.g., `0x01`) |
| 5 | 3B | Flags | Reserved (Compression, Encryption) |

#### 4.1.2. Entry Frame Structure (Repeated)

| Size | Field | Validation / Purpose |
| :--- | :--- | :--- |
| **4B** | **Sync Marker** | Fixed constant `0xFA 0xCE 0xB0 0x0C` for SIMD scanning. |
| **4B** | **Length (N)** | Size of Payload. |
| **4B** | **~Length** | Bitwise NOT of Length. Used for **FEC**. If `Len != ~InvLen`, header is corrupt. |
| **1B** | **Type** | `0x01`=StandardLog, `0x02`=Metric, `0x03`=Trace. |
| **1B** | **Header CRC** | CRC-8 of the preceding 10 bytes (Len, ~Len, Type). **Prevents massive seeking on bit-flips.** |
| **N** | **Payload** | Normalized MessagePack binary data. |
| **4B** | **Payload CRC** | CRC-32 of the Payload. Ensures data integrity. |

### 4.2. Compaction Strategy

The Compactor runs as a background `IHostedService`.

1. **Trigger:** Time-based (every 10m) or Size-based (WAL > 100MB).
2. **Reader Logic:**
    * Scans WAL using SIMD to find `SyncMarker`.
    * Validates Header CRC. If failed $\to$ Skip 1 byte $\to$ Scan again (Re-sync).
    * Validates Payload CRC. If failed $\to$ Log Error $\to$ Skip Entry $\to$ Scan for next Marker.
3. **Schema Resolution (The "Union" Approach):**
    * The Compactor samples the batch.
    * **Conflict Resolution:**
        * `Key: Int` vs `Key: String` $\to$ Promote column to `String`.
        * `Key: Float` vs `Key: Int` $\to$ Promote column to `Double`.
    * **Overflow:** Rare or highly dynamic keys are packed into a single `_meta` JSON column to prevent Parquet column explosion.
4. **Output:** Writes a Parquet file named idempotently: `{Stream}_{MinTimestamp}_{MaxTimestamp}_{Hash}.parquet`.
5. **Checkpoint:** Updates `cursor.json` only after the Parquet file is successfully flushed.

### 4.3. Search Implementation

**DuckDB** is used to query the Parquet files directly.

* **Virtual Columns:** The system maps the canonical `_t` (timestamp) to Parquet's generic timestamp type for predicate pushdown optimization.
* **Query Translation:**
  * Request: `?q=level='ERR' AND duration > 500`
  * SQL: `SELECT * FROM read_parquet('stream/*.parquet') WHERE level = 'ERR' AND duration > 500`

## 5. Non-Functional Requirements (NFRs)

### 5.1. Performance

* **Ingestion Throughput:** Target > 500 MB/s per node (leveraging `System.IO.Pipelines` and Kestrel).
* **Allocations:** Zero-copy ingestion where possible. Use `Span<byte>` and `ArrayPool` for WAL writing.
* **Startup Time:** < 2 seconds, even with uncompacted WALs (enabled by sequential WAL scanning).

### 5.2. Reliability

* **Durability:** `FileOptions.WriteThrough` enabled on the WAL to survive OS crashes.
* **Corruption Tolerance:** The system must **never** crash due to a corrupt WAL file. It must skip the bad frame and self-correct using the Sync Marker.
* **Backward Compatibility:** The system must be able to read WAL files generated by previous versions of the software (via Versioned Reader Factory).

### 5.3. Observability (Meta-Monitoring)

* The system must expose metrics:
  * `wal_corruptions_detected` (Counter)
  * `ingestion_rate_bytes` (Rate)
  * `compaction_lag_ms` (Gauge)

## 6. Implementation Plan (Phased)

### Phase 1: Core Storage (The "Spine")

* Implement `WalWriter` with the Resilient Frame format.
* Implement `WalReader` with SIMD scanning and CRC validation.
* Create basic Ingestion API (JSON only).

### Phase 2: Compaction & Parquet

* Implement the background Compactor service.
* Integrate `Apache.Arrow` or `Parquet.Net` for writing files.
* Handle basic schema evolution (Append columns).

### Phase 3: Query & OTel

* Integrate DuckDB.NET.
* Add OTLP (Protobuf) endpoint.
* Implement the "Normalization" layer to convert OTel $\to$ Internal Binary.

### Phase 4: Hardening

* Implement L2 Compaction (Daily merge).
* Fuzz testing: Intentionally corrupt WAL files (random bit flips) and verify `WalReader` recovery.

## 7. Open Questions / Decision Points

1. **Compression in WAL?** Currently disabled for speed. Should we enable LZ4 for the WAL payload if network I/O becomes a bottleneck?
    * *Decision:* No, keep WAL raw for maximum write speed. Compress only at Parquet level.
2. **Retention Policy:** How strictly do we delete old Parquet files?
    * *Decision:* Configurable per stream (TTL in days).
3. **Indexing:** Do we need a sidecar index (Bloom Filter) for high-cardinality fields like `TraceId`?
    * *Future:* DuckDB handles brute-force scanning fast enough for now, but Bloom Filters might be needed for petabyte scale.
